# CODEX Next Experiment Recommendation

## 1) Exact next experiment to run

## **Run a real GSM8K paired-strategy data collection on 100 queries** (not synthetic), using exactly:

- `reasoning_greedy`
- `direct_plus_revise`

and build one query-level routing table with:

- `question_id`, `question`, `gold_answer`
- `reasoning_answer`, `revise_answer`
- `reasoning_correct`, `revise_correct`
- `revise_helpful = 1 iff reasoning_correct==0 and revise_correct==1`
- currently implemented engineered features

Use the existing pipeline entry point:

```bash
python3 scripts/run_build_real_routing_dataset.py --subset-size 100
```

If blocked, emit blocker summary only (already implemented behavior).

## 2) Why this is higher priority than other possible next steps

This is the minimal experiment that directly answers the central scientific bottleneck:

- Does revise-worthiness occur often enough on real GSM8K?
- Do we have real labels to train/evaluate routing?
- What are baseline accuracy/cost characteristics of cheap vs corrective strategy?

Without this dataset, any learned routing result is impossible, and adding more heuristics/policies/strategies is mostly ungrounded engineering.

## 3) Required inputs/data

Required:

- `data/gsm8k_uploaded_normalized.jsonl` (already generated by validator)
- Working `OPENAI_API_KEY`
- Accessible model backend for runner default (`gpt-4o-mini`) or explicitly configured alternative

Optional for follow-up ML step (not required for the collection step itself):

- `scikit-learn`

## 4) Exact outputs/artifacts it should produce

Primary artifacts from the real collection run:

- `outputs/real_routing_dataset/gsm8k_subset_run_summary.json`
- `outputs/real_routing_dataset/gsm8k_per_query_outputs.csv`
- `data/real_gsm8k_routing_dataset.csv`

Minimum summary fields to inspect immediately:

- run status
- subset executed
- `revise_helpful_positives` and rate
- `reasoning_greedy` accuracy proxy (`reasoning_correct` mean)
- `direct_plus_revise` accuracy proxy (`revise_correct` mean)

## 5) What conclusion it would allow us to make

This one run would allow a real, non-synthetic statement about whether revise-worthiness exists at actionable prevalence on GSM8K and whether corrective strategy meaningfully outperforms cheap strategy enough to motivate learned routing.

Concretely, it enables:

- measured-now estimate of label prevalence (`revise_helpful` rate)
- measured-now baseline gap between `reasoning_greedy` and `direct_plus_revise`
- a decision on whether lightweight ML routing is justified next

## 6) What remains unresolved after it

Even after this run, unresolved items remain:

- statistical stability (100 queries may still be noisy)
- generalization to MATH500/harder domains
- final claim strength for paper-level conclusions
- comparison to external baselines (TALE/BEST-Route)

If label prevalence is near-zero or near-one, the project may need to rethink route definition or correction strategy; if prevalence is moderate and features correlate, then lightweight routing ML is the immediate next step.
